# CLAUDE.md

Guía para Claude Code (claude.ai/code) cuando trabaja en este repositorio.

Documento VIVO: cuando el proyecto cambie de stack, comandos o reglas, actualízalo en el
mismo commit que el cambio. Un CLAUDE.md desactualizado es peor que ninguno, porque el agente
confiará en información falsa. El espejo para agentes no-Claude es `AGENTS.md` (mantener sincronizado).

**Proyecto:** CRM Pulse
**Versión del documento:** 1.0.0

---

## Reglas Absolutas

1. **MODO PLAN PARA TAREAS NO TRIVIALES**: antes de ejecutar cualquier tarea de 3+ pasos o que
   implique una decisión de arquitectura, presentar un plan. En este proyecto el plan se muestra
   **EN PANTALLA, dentro del chat (no en un archivo)** y se espera el veredicto **GO** del equipo
   de auditoría del dueño antes de implementar código. Un **NO-GO** → incorporar hallazgos
   (bloqueantes/mayores) → volver a mostrar el plan actualizado, iterando hasta el GO. Si algo se
   rompe durante la ejecución, PARAR y replanificar antes de continuar.
2. **BUCLE DE MEJORA CONTINUA**: después de CUALQUIER error en una sesión, registrarlo de
   inmediato en `.claude/memory/gotchas.md` con fecha, categoría, causa raíz y regla preventiva.
   Al inicio de cada sesión, revisar `gotchas.md` junto con `.claude/memory/MEMORY.md`. Cuando un
   patrón alcance 3+ ocurrencias, promoverlo a «Errores Críticos» en este archivo con ejemplo de
   código. Si es sistémico, crear una Regla Absoluta nueva.
3. **ESTRATEGIA DE SUBAGENTES**: para problemas complejos, usar subagentes agresivamente. Dividir
   en tareas de investigación, ejecución y análisis. Una tarea por agente. Lanzar varios en
   paralelo cuando sean independientes.
4. **VERIFICACIÓN ANTES DE DAR POR HECHO**: NUNCA marcar una tarea como completada sin prueba
   verificable. Correr los checks (typecheck, build, tests) y, para cambios de UI o de flujo,
   ejercitar el comportamiento real (skill `verify` / `run`). El typecheck y el build NO validan
   la experiencia. Comparar comportamiento esperado contra actual.
5. **IDIOMA CONTEXTUAL**: comunicación operativa, commits, documentación y **dominio de producto**
   en **español**. Los identificadores de dominio van en español coherente con el producto y el
   diseño (`cliente`, `seguimiento`, `propietario`, `resolverUsuario`, `calcularIntervalos`). Los
   nombres de framework/librería/API se dejan en su forma original (hooks de React, API de Convex,
   `useQuery`, etc.). No traducir lo que el ecosistema define en inglés.
6. **SIN REFERENCIAS A IA**: JAMÁS incluir en commits, código, documentación o PRs referencias a
   la IA que ayudó a producirlos. Sin "Co-Authored-By", sin "Generated by". El trabajo se firma
   como del dueño. (Documentar un producto de terceros que el proyecto integra —p. ej. la API de
   un proveedor— es contenido factual y sí está permitido.)
7. **CORREGIR LO PROPIO, REPORTAR LO PREEXISTENTE**: todo fallo introducido por el trabajo en
   curso (build, lint, types, lógica, seguridad) se corrige dentro del mismo cambio. La deuda o
   los bugs preexistentes que aparezcan se REPORTAN con detalle (archivo, línea, impacto) sin
   ampliar el alcance por cuenta propia.
8. **POLÍTICA DE DECISIONES**: NUNCA decidir stack, arquitectura o dependencias mayores sin
   aprobación explícita del dueño. Presentar opciones con pros y contras y esperar la decisión.
   Acciones autónomas permitidas: corrección de bugs, refactor menor, actualización de
   dependencias menores.
9. **NUNCA HACER PUSH SIN APROBACIÓN EXPLÍCITA**: **en este proyecto `git push origin main`
   equivale a desplegar a producción en Railway** (auto-deploy). No se sube nada al remoto sin
   permiso directo del dueño. Commits locales sí están permitidos. Sin excepciones.
10. **FLUJO GIT — TRABAJO EN `main`**: la rama de trabajo y de deploy es **`main`** (Railway
    auto-despliega desde ahí; NO hay flujo de PR ni otra rama de larga vida en este proyecto). Commits locales
    frecuentes con Conventional Commits (`feat:`, `fix:`, `refactor:`, `docs:`, `chore:`). Ramas
    `feat/*` opcionales para trabajo experimental, con merge a `main`. El push a `main` solo
    con aprobación (regla 9).
11. **SYNC DEL REPO AL INICIO**: `git fetch origin` al comenzar cada sesión cuando el repo tenga
    historia compartida con el remoto (repo de un solo dev en una máquina → normalmente
    innecesario, pero no hace daño).
12. **VALIDAR ANTES DE PUSH**: antes de cualquier push a `main`, ejecutar localmente TODOS los
    Quality Gates. Push = deploy: no subir código que vaya a fallar en build o en runtime.
13. **NUNCA BAJAR VERSIONES DE DEPENDENCIAS**: sin downgrade sin aprobación. Si un paquete causa
    problemas, buscar solución compatible con la versión actual o superior.
14. **VERSIONES DE DEPENDENCIAS**: preferir la última versión estable al añadir algo nuevo
    (`npm view <pkg> dist-tags`), PERO **respetar los pins que exige el ecosistema**. Ejemplo
    real: Convex Auth ancla `@auth/core` a una versión exacta (~`0.41.1`); usar "latest" ahí
    rompe la autenticación. La compatibilidad manda sobre "lo más nuevo".
15. **SKILLS OBLIGATORIOS**: antes de implementar, modificar o debuggear en un dominio con skill
    (ver «Skills por Dominio»), CARGAR el skill correspondiente.
16. **SIN SECRETS EN EL REPO**: nunca commitear claves, tokens ni credenciales. Los secretos
    viven en `.env.local` (gitignored) o en las variables del deployment (Convex / Railway). Un
    secreto filtrado se ROTA de inmediato y se purga del historial.
17. **AUTORIZACIÓN EN CADA FUNCIÓN (modelo Convex, NO RLS)**: Convex **no tiene** seguridad a
    nivel de fila (RLS es de Postgres/Supabase). En su lugar: TODA `query`/`mutation` valida la
    identidad y autoriza **dentro de sí misma** (patrón `requireUsuario` / `resolverUsuario`);
    nunca se confía en el cliente para decisiones de acceso. Los datos se validan con validadores
    en `convex/schema.ts`. Fallar **cerrado** siempre (nunca `identidad ?? dev`).
18. **VERIFICAR UI DE PLATAFORMAS ANTES DE INSTRUIR**: antes de dictar pasos de una interfaz
    externa (dashboards de Convex, Railway, Linear), verificar el estado actual con documentación
    oficial reciente o una captura del usuario. Las UIs cambian sin aviso. Si no se puede
    verificar, decirlo.
19. **GESTIÓN DEL TRABAJO VÍA LINEAR**: el trabajo (feature, fix, refactor) debería tener una
    issue en **Linear** (equipo Talent Academy Curso `TAL`, proyecto "PROY CRM Pulse") en estado
    **"In Progress"** antes de tocar código, y pasar a **"Done"** al terminar. Nota: el equipo TAL
    NO tiene estado "In Review" (estados: Backlog/Todo/In Progress/Done/Canceled/Duplicate). Leer
    archivos o responder preguntas no es "trabajo": no requiere issue.
20. **PREGUNTAS AL DUEÑO**: agrupar las decisiones dentro de una misma revisión de plan (se
    presentan juntas para que el equipo de auditoría las resuelva en bloque). Las dudas sueltas
    fuera de un plan se hacen de **una en una** (pregunta → respuesta → siguiente) para no mezclar
    respuestas.

## Reglas específicas de CRM Pulse

- **Fuente de verdad de producto = el diseño.** Las 11 pantallas viven en
  `design/PROY CRM Pulse/*.dc.html`. El MVP = esas pantallas construidas al 100% con sus datos y
  filtros. Ante duda de producto, el diseño manda.
- **Derivados NO se persisten**: el *estado* del cliente (nuevo_lead/en_negociación/ganado/perdido),
  su *valor* (suma de ventas ganadas), el *último contacto* y el *total* de una venta se **calculan
  en queries**, no se guardan en la tabla.
- **Borrado = archivar (soft-delete)** vía campo `archivado`. Las listas y los derivados ignoran
  los archivados.
- **`convex/_generated` SE VERSIONA** (el build de Railway no depende de regenerarlo).
- **Los tests de Convex se excluyen del typecheck de Convex** (`convex/tsconfig.json`); si no,
  `import.meta.glob` rompe el push de `convex dev`.
- **Env-gates de desarrollo**: `CRM_DEV_USER_EMAIL` (fallback de usuario dev, solo cuando NO hay
  identidad) y `CRM_ALLOW_SEED=true` (semillas). Ausentes en producción → fallan cerrado.
- **Deploy**: front/app en **Railway** desde `main`; funciones Convex vía `npx convex deploy`.
  Ver `DEPLOY.md`. Requiere `NEXT_PUBLIC_CONVEX_URL` (se inlinea en build).

## Descripción del Proyecto

**CRM Pulse** es un CRM para negocios pequeños de ventas: ayuda a no perder ventas por falta de
seguimiento. Gestiona clientes (contactos), interacciones, seguimientos (con una "Agenda del día"
de tareas vencidas/hoy/próximas) y ventas/oportunidades. Multi-vendedor: cada cliente tiene un
propietario y cada venta un vendedor; roles `dueña` y `vendedor`. Producto en español.

**Estado:** MVP en construcción. M0 (diseño) completo; M1 (fundación técnica) + M4.3 (Agenda del
día) + shell de navegación implementados y desplegados en Railway. En curso: M2 (login/auth real
con Convex Auth). Siguen M3 (clientes), M4 (seguimiento), M5 (ventas), M6 (cierre).

### Convenciones de Código

- **Nombres**: componentes `PascalCase.tsx`; helpers/módulos `kebab-case` o `camelCase`; funciones
  y variables `camelCase`. Dominio de producto en español (ver regla 5); framework/API en inglés.
- **Imports**: usar el alias `@/…` en lugar de rutas relativas profundas (`../../../`).
- **Comentarios**: solo para explicar lo que el código no dice por sí mismo (una restricción, un
  porqué). Nunca para narrar lo obvio. En español.
- **Convex**: IDs `Id<"tabla">` (no uuid); acceso a datos vía `query`/`mutation` + hooks
  reactivos (`useQuery`/`useMutation`); autorización dentro de cada función (regla 17).

## Errores Críticos — NO Repetir

> Cuando un patrón se repita 3+ veces en `gotchas.md`, se promueve aquí con ejemplo. Los de abajo
> son restricciones establecidas del proyecto.

### 1. Autenticación que falla ABIERTA (`identidad ?? dev`)

Resolver el usuario con un fallback que se activa aunque haya identidad abre la puerta a heredar
una cuenta ajena. Debe fallar **cerrado**.

```ts
// INCORRECTO — si la identidad no está aprovisionada, hereda el usuario dev
return identidad ?? usuarioDev

// CORRECTO — identidad presente sin usuario → null; el fallback dev SOLO sin identidad
if (identidad) return await buscarUsuario(identidad)   // puede ser null
if (process.env.CRM_DEV_USER_EMAIL) return await buscarDev()
return null
```

### 2. Tests de Convex rompen el push de `convex dev`

`import.meta.glob` en los `*.test.ts` hace fallar el typecheck de `convex dev`. Mantener la
exclusión en `convex/tsconfig.json` (`"exclude": ["./_generated", "./**/*.test.ts"]`).

## Comandos Esenciales

```bash
# Desarrollo
npm run dev                    # Next.js dev (http://localhost:3000)
npx convex dev                 # Convex en watch (genera _generated y NEXT_PUBLIC_CONVEX_URL en .env.local)

# Validación local (ANTES de commit / push) — ver «Quality Gates»
npx tsc --noEmit               # typecheck de la app (no hay script dedicado)
npm run lint                   # next lint
npm run build                  # next build (valida tipos + compila)
npm run test                   # vitest run
npx convex dev --once          # empuja esquema/funciones y typecheck de Convex

# Commit / deploy
git add -A && git commit -m "feat: descripción"   # commits locales OK
git push origin main                            # SOLO con aprobación (regla 9) → Railway auto-deploya
```

## Quality Gates

Todo esto en verde antes de un push a `main` (= deploy):

- `npx tsc --noEmit` + `npm run lint` + `npm run build` sin errores.
- `npm run test` (vitest) cubriendo el cambio.
- `npx convex dev --once` sin errores de esquema/typecheck de Convex.
- Autorización verificada dentro de cada función Convex nueva/modificada (regla 17).
- Para cambios de UI: verificación manual en navegador — escritorio (≥1280px) y móvil (375px) —
  con el skill `verify`/`run`.

```text

[ ] Tests del cambio en verde (vitest)
[ ] Convex: `npx convex dev --once` limpio; _generated regenerado y versionado
[ ] Autorización dentro de cada función Convex nueva (no RLS; falla cerrado)
[ ] No hay secrets en el diff
[ ] UI validada en escritorio y móvil si el cambio toca UI/CSS/layout
[ ] Skill correspondiente cargado antes de tocar su dominio
[ ] Commit en Conventional Commits, en español, SIN referencias a IA

[ ] Push a `main` solo con aprobación explícita del dueño
```
