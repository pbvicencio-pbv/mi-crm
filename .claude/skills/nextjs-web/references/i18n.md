## 1. Setup

### Install

```bash
pnpm add next-intl
```

### Configuration files

```ts
// i18n/routing.ts — define supported locales and routing behavior
import { defineRouting } from 'next-intl/routing'

export const routing = defineRouting({
  locales: ['en', 'es', 'pt'] as const,
  defaultLocale: 'en',
  localePrefix: 'always',           // 'always' | 'as-needed' | 'never'
  pathnames: {
    '/about': { en: '/about', es: '/acerca', pt: '/sobre' },
    '/feed/[itemId]': { en: '/feed/[itemId]', es: '/contenido/[itemId]', pt: '/conteudo/[itemId]' },
  },
})

export type Locale = (typeof routing.locales)[number]
export type Pathnames = typeof routing.pathnames
```

```ts
// i18n/request.ts — configure messages loading
import { getRequestConfig } from 'next-intl/server'
import { routing } from './routing'

export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale
  if (!locale || !routing.locales.includes(locale as any)) {
    locale = routing.defaultLocale
  }

  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default,
  }
})
```

```ts
// i18n/navigation.ts — typed navigation helpers
import { createNavigation } from 'next-intl/navigation'
import { routing } from './routing'

export const { Link, redirect, usePathname, useRouter, getPathname } =
  createNavigation(routing)
```

### next.config.ts integration

```ts
import createNextIntlPlugin from 'next-intl/plugin'

const withNextIntl = createNextIntlPlugin('./i18n/request.ts')

const nextConfig = {
  cacheComponents: true,
  // ... rest of config
}

export default withNextIntl(nextConfig)
```

### Type registration (v4.0+)

```ts
// global.d.ts
import type { routing } from './i18n/routing'
import type messages from './messages/en.json'

declare module 'next-intl' {
  interface AppConfig {
    Locale: (typeof routing.locales)[number]
    Messages: typeof messages
  }
}
```

This enables: strict locale typing, compile-time message key validation, and autocomplete for translation keys.

## 2. Routing Strategies

### Prefix-based (recommended)

Every URL has a locale prefix: `/en/feed`, `/es/feed`.

```ts
defineRouting({
  locales: ['en', 'es'],
  defaultLocale: 'en',
  localePrefix: 'always',  // All URLs have prefix, including default locale
})
```

Variant: `localePrefix: 'as-needed'` omits prefix for default locale (`/feed` = English, `/es/feed` = Spanish).

### Domain-based

```ts
defineRouting({
  locales: ['en', 'es'],
  defaultLocale: 'en',
  domains: [
    { domain: 'example.com', defaultLocale: 'en' },
    { domain: 'example.es', defaultLocale: 'es' },
  ],
})
```

### Localized pathnames

Different URL slugs per locale:

```ts
pathnames: {
  '/about': { en: '/about', es: '/acerca' },
  '/feed': { en: '/feed', es: '/contenido' },
}
```

URL: `/en/about` → `/es/acerca`

### proxy.ts locale detection

```ts
// proxy.ts
import createMiddleware from 'next-intl/middleware'
import { routing } from './i18n/routing'

export const proxy = createMiddleware(routing)

export const config = {
  matcher: ['/', '/(en|es|pt)/:path*'],
}
```

Note: next-intl still exports `createMiddleware` — it works in `proxy.ts` despite the name. The function detects locale from cookies, Accept-Language header, or URL, and redirects accordingly.

## 3. Server Components

### `getTranslations()` — async, zero client JS

```tsx
// app/[locale]/(app)/feed/page.tsx — Server Component
import { getTranslations } from 'next-intl/server'

export default async function FeedPage() {
  const t = await getTranslations('feed')

  return (
    <div>
      <h1>{t('title')}</h1>
      {/* All rendered on server — no JS sent to client */}
    </div>
  )
}
```

### `getFormatter()` — formatting on the server

```tsx
import { getFormatter } from 'next-intl/server'

export default async function ItemPage({ item }) {
  const format = await getFormatter()

  return (
    <time>{format.dateTime(new Date(item.createdAt), { dateStyle: 'long' })}</time>
  )
}
```

### Metadata with translations

```tsx
import { getTranslations } from 'next-intl/server'

export async function generateMetadata({ params }: { params: Promise<{ locale: string }> }) {
  const { locale } = await params
  const t = await getTranslations({ locale, namespace: 'feed' })

  return {
    title: t('title'),
    description: t('description'),
  }
}
```

## 4. Client Components

### `useTranslations()` — interactive content

```tsx
'use client'
import { useTranslations } from 'next-intl'

export function FeedFilters() {
  const t = useTranslations('feed.filters')

  return (
    <div className="flex gap-2">
      <button>{t('all')}</button>
      <button>{t('tech')}</button>
      <button>{t('design')}</button>
    </div>
  )
}
```

### Typed navigation

```tsx
'use client'
import { Link, useRouter } from '@/i18n/navigation'

export function NavLink() {
  const router = useRouter()

  return (
    <>
      {/* Typed Link — autocompletes pathnames */}
      <Link href="/feed">Feed</Link>
      <Link href={{ pathname: '/feed/[itemId]', params: { itemId: '123' } }}>Item</Link>

      {/* Programmatic navigation */}
      <button onClick={() => router.push('/search')}>Go to Search</button>
    </>
  )
}
```

### `useLocale()` — current locale

```tsx
'use client'
import { useLocale } from 'next-intl'

export function LocaleSwitcher() {
  const locale = useLocale()
  // ...
}
```

## 5. Formatting

### Date and time

```tsx
const format = await getFormatter()

format.dateTime(date, { dateStyle: 'long' })                    // February 23, 2026
format.dateTime(date, { dateStyle: 'short', timeStyle: 'short' }) // 2/23/26, 3:45 PM
format.relativeTime(date)                                         // 2 hours ago
format.relativeTime(date, { now: referenceDate })                 // in 3 days
```

### Numbers and currency

```tsx
format.number(1234.56)                                    // 1,234.56
format.number(1234.56, { style: 'currency', currency: 'EUR' })  // €1,234.56
format.number(0.85, { style: 'percent' })                 // 85%
```

### Lists

```tsx
format.list(['React', 'Next.js', 'TypeScript'], { type: 'conjunction' })
// → "React, Next.js, and TypeScript"
```

## 6. Pluralization and ICU

next-intl supports full ICU MessageFormat syntax:

```json
{
  "search": {
    "results": "{count, plural, =0 {No results} one {1 result} other {{count} results}} for \"{query}\""
  },
  "notifications": {
    "unread": "You have {count, plural, =0 {no unread notifications} one {1 unread notification} other {{count} unread notifications}}"
  }
}
```

```tsx
t('search.results', { count: 42, query: 'react' })
// → '42 results for "react"'

t('notifications.unread', { count: 0 })
// → 'You have no unread notifications'
```

### Select (gender, categories)

```json
{
  "greeting": "{gender, select, male {He} female {She} other {They}} liked your post"
}
```

## 7. SEO and Metadata

### `hreflang` links

next-intl automatically adds `hreflang` link tags when using its routing system. Verify in HTML output:

```html
<link rel="alternate" hreflang="en" href="https://example.com/en/feed" />
<link rel="alternate" hreflang="es" href="https://example.com/es/contenido" />
<link rel="alternate" hreflang="x-default" href="https://example.com/en/feed" />
```

### Localized sitemap

```ts
// app/sitemap.ts
import { routing } from '@/i18n/routing'
import { getPathname } from '@/i18n/navigation'

export default async function sitemap() {
  const routes = ['/', '/feed', '/about']

  return routes.flatMap((route) =>
    routing.locales.map((locale) => ({
      url: `https://example.com${getPathname({ locale, href: route })}`,
      lastModified: new Date(),
      alternates: {
        languages: Object.fromEntries(
          routing.locales.map((l) => [l, `https://example.com${getPathname({ locale: l, href: route })}`])
        ),
      },
    }))
  )
}
```

### `<html lang>` per locale

```tsx
// app/[locale]/layout.tsx
export default async function LocaleLayout({ children, params }) {
  const { locale } = await params
  return (
    <html lang={locale}>
      <body>{children}</body>
    </html>
  )
}
```

## 8. Comparison: next-intl vs Paraglide

| Criterion | next-intl 4.8 | Paraglide JS 2.12 |
|---|---|---|
| **Approach** | Runtime ICU interpolation | Compile-time function generation |
| **Client bundle** | ~2 KB + messages loaded | ~0 KB (per-message tree-shaking) |
| **Server Components** | ✅ Native `getTranslations()` | ✅ Direct function import |
| **Routing** | ✅ Prefix, domain, localized pathnames | ✅ Prefix-based |
| **Date/number formatting** | ✅ Built-in `useFormatter()` | ❌ Use native `Intl` APIs |
| **Pluralization** | ✅ Full ICU MessageFormat | ✅ Via compile-time functions |
| **Dynamic keys** | ✅ `t(dynamicKey)` | ❌ Keys must be static |
| **Type safety** | ✅ AppConfig + augmented types | ✅ Generated typed functions |
| **npm downloads** | ~1M weekly | ~40K weekly |
| **Ecosystem maturity** | Production-proven, extensive docs | Growing, smaller community |
| **Proxy/middleware** | ✅ `createMiddleware()` built-in | ✅ Compiler plugin |

### When to choose Paraglide

Choose Paraglide only when: (1) bundle size is the absolute #1 priority, (2) you don't need built-in date/number formatting, (3) you're comfortable with static-only keys, and (4) you don't need the routing integration that next-intl provides. For most production App Router projects, next-intl is the better choice.
